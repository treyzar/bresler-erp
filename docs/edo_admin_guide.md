# ЭДО — гайд администратора

Для пользователей в группе `admin`. Меню сайдбара под **ЭДО →
Администрирование** содержит три раздела: «Типы документов»,
«Шапки организаций», «Отчёты».

## 1. Типы документов

`/edo/admin/types`

Список + кнопка **Создать тип**.

### Создание типа

**Основное.**
- **Код** (slug, snake_case) — иммутабельный, используется в URL и API.
- **Название** — отображается в каталоге.
- **Категория** — `memo` / `application` / `notification` / `travel` /
  `bonus` / `other`. Влияет на группировку в каталоге.
- **Иконка** — имя [lucide-иконки](https://lucide.dev/icons/).
- **Нумерация** — `NumberSequence` из справочника core (создаются в
  Django admin или через миграции).
- **Видимость** — `personal_only` (только автор + согласующие) /
  `department_visible` (видят сотрудники подразделения автора) /
  `public` (видят все, в рамках tenant-скоупа).
- **Кто может создавать** — `author` (любой) / `department_head` /
  `group:accounting` / `group:hr` / `group:admin`.
- **Требует рисованную подпись** — флаг для документов, у которых на
  последнем шаге согласующий должен расписаться мышкой/пальцем (cmiety,
  vacation, travel).

**Поля формы (`field_schema`).**
Конструктор полей: добавление/удаление, перестановка кнопками ↑/↓.
Для каждого поля — имя (snake_case), подпись, тип, обязательность, help_text.

Conditional подразделы:
- **type=choice** → редактор вариантов (`code → label`).
- **type=table** → редактор колонок (имя, заголовок, тип ячейки).
  Внутри таблицы можно использовать любые типы, кроме вложенных таблиц.

**Шаблоны.**
- **`title_template`** — Django Template Language для генерации `Document.title`.
- **`body_template`** — DTL для тела документа. Доступны переменные:
  - `author` (User: `author.full_name`, `author.position`, `author.department_unit.name`),
  - `today` (date),
  - `document` (после submit: `document.number`, `document.created_at`),
  - `fields` (dict с гидрированными значениями) + каждое поле по имени.
  - Для `choice` доступна также `<name>_display` с человекочитаемой меткой.
  - Для `table` — список dict'ов с гидрированными User/Department объектами.

**Цепочка согласования.**
Список шагов с reorder ↑/↓. Для каждого:
- **role_key** — preset (Непосредственный руководитель / Бухгалтерия и т.д.)
  или ручной ввод. Полный список префиксов:
  | Синтаксис | Что |
  |---|---|
  | `supervisor` | Руководитель автора (по `User.supervisor` или дереву Department) |
  | `dept_head:self` / `dept_head:parent` / `dept_head:up(N)` | Head of dept-узла относительно автора |
  | `company_head` | Head компании автора (`is_department_head=True, department_unit=NULL`) |
  | `group:NAME` / `group:NAME@company` | Любой активный участник группы (с/без скоупа компании) |
  | `group_head:NAME` | Участник группы с `is_department_head=True` |
  | `fixed_user:42` | Конкретный User по pk |
  | `field_user:NAME` | User из FK-поля `NAME` в `field_values` |
  | `field_user_supervisor:NAME` | Руководитель того, на кого указывает поле `NAME` |
  | `field_dept_head:NAME` | Head of Department из поля `NAME` |
  | `author` | Сам автор (для шагов подписи входящих уведомлений) |
- **Действие** — `approve` (требует одобрения) / `sign` (то же + подпись) /
  `inform` / `notify_only` (без блокировки, авто-завершаются при активации
  следующего active-шага).
- **SLA (часы)** — после этого срока шаг помечается просроченным,
  отправляются эскалационные уведомления.
- **Параллельная группа** — текстовый id; все шаги одной группы становятся
  активными одновременно.
- **Режим параллели** — `AND` (требуется решение всех) / `OR`
  (достаточно одного approve, остальные → SKIPPED; reject в OR не блокирует).

После сохранения новый тип доступен пользователям с подходящим
`initiator_resolver` в каталоге `/edo/new/`.

### Редактирование

При редактировании можно менять всё, кроме `code`. Изменения **не
влияют на уже отправленные документы** — у них зафиксирован
`chain_snapshot` на момент submit. Только новые документы будут
использовать обновлённый шаблон/цепочку.

## 2. Шапки организаций

`/edo/admin/org-heads`

Простой CRUD-список: организация + ФИО руководителя + должность +
период действия (`from_date`, `to_date`).

При создании документа в `Document.header_snapshot` пишется ФИО и
должность активного на эту дату руководителя. Это выводится в шапку
PDF в блоке «Кому». Смена директора задним числом не ломает архив.

### Несколько руководителей

Можно завести цепочку записей с непересекающимися периодами — в архиве
останется правильный директор для каждого документа. Текущая запись
(без `to_date` или с `to_date >= today`) считается активной.

## 3. Отчёты и массовые операции

`/edo/admin/reports`

### Экспорт архива (ZIP)

Кнопка **Скачать ZIP** с фильтром по периоду. В архиве:
- `index.json` — сводный список;
- по каждому документу: `metadata.json` + `document.pdf` + папка
  `attachments/`.

В response-заголовке `X-Archive-Summary` — статистика по успешным/
неудачным PDF и числу вложений.

### Висящие документы (≥ 3 дней)

Документы в `PENDING` дольше 3 дней. По строкам — чекбоксы для bulk-операций:
- **Напомнить** — отправить уведомление текущим согласующим (комментарий
  опционален, попадёт в текст уведомления).
- **Отменить** — принудительно закрыть выбранные документы как
  `cancelled` (комментарий **обязателен** — попадёт в историю шагов
  как причина отмены).

Это override обычного `cancel`: разрешено даже если уже есть одобрения.
В финальном статусе (approved/rejected/cancelled) — пропускается.

### Нарушения SLA за 30 дней

Шаги, где `sla_breached_at` зафиксирован Celery-задачей. Сама задача
запускается раз в час; повторно один и тот же шаг не уведомляет.

### Топ типов за 30 дней

Гистограмма: type → total / approved / rejected / pending. Используется
для понимания нагрузки и ROI на конкретные типы.

## Управление группами и правами

Базовая модель — Django Groups. Для EDO значимы:
- `admin` — доступ ко всему `/edo/admin/...`, может создавать любой тип
  документа, видит все документы в `for_user`.
- `accounting` — резолвится в шагах `group:accounting` и `group:accounting@company`,
  обязательная для создателей `vacation_notification`.
- `hr` — для будущих кадровых типов; используется в `group:hr` resolver.

Группа создаётся через Django admin (`/admin/auth/group/`). Подключение
сотрудника — там же (или из админки User).

### Замещения

Поля `User.substitute_user`, `substitute_from`, `substitute_until`. На
момент **активации** шага (WAITING → PENDING) проверяется, в отпуске
ли назначенный approver: если да, шаг автоматически переадресуется
замещающему, оригинал кладётся в `original_approver`.

> ⚠️ Уже PENDING-шаги не перерезолвятся при включении замещения задним
> числом — это by design (иначе нарушится аудит). Для открытых шагов
> используй кнопку «Делегировать» вручную.

## Celery Beat задачи

Запланированные периодически:
| Задача | Период | Что |
|---|---|---|
| `edo-cleanup-pdf-cache` | 24h | Удаляет файлы кеша PDF старше TTL (`InternalDocFlowConfig.pdf_cache_ttl_hours`, дефолт 168ч/7д) |
| `edo-check-sla-breaches` | 1h | Сканирует pending active-шаги с просроченным `sla_due_at`; помечает + триггерит уведомления |

Глобальные настройки модуля — singleton `InternalDocFlowConfig` в Django
admin (`/admin/internal_docs/internaldocflowconfig/`):
- `cross_company_scope` — `company_only` (default) / `group_wide` —
  определяет видимость публичных типов между компаниями группы.
- `default_sla_hours` — fallback SLA для шагов без явного значения (default 48).
- `pdf_cache_ttl_hours` — TTL on-demand PDF-кеша (default 168 = 7 дней).

## Troubleshooting

| Симптом | Причина | Решение |
|---|---|---|
| Документ при submit отдаёт 400 «Не удалось собрать цепочку» | role_key не резолвится — например, `supervisor` для одинокого пользователя без head'а в дереве | Назначить `is_department_head=True` нужному пользователю или прописать `User.supervisor` явным FK |
| Не приходит email-уведомление | `EMAIL_HOST_*` не настроены или у пользователя в `NotificationPreference` отключён email-канал по этой категории | Проверить env + `/api/notifications/preferences/` |
| Шаг group `inbox_for` не показывает документ нужным сотрудникам | Не совпадает скоуп `@company` или сотрудник не в нужной группе | Проверить `User.groups`, `User.company_unit`; для `group:NAME@company` author и approver должны быть в одной компании |
| PDF не генерится | Playwright не установлен / нет браузерного бинарника | См. `apps/edo/templates_app/services/pdf_export.py` — там код запуска; на сервере нужен `playwright install chromium` |
