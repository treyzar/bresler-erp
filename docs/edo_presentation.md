# Электронный документооборот

## Внутренние документы предприятия — без бумаги, без потерь, с прозрачной историей

<br>

**Bresler ERP** · модуль ЭДО · апрель 2026

---

## Зачем это нужно

<br>

<div class="lead">

Внутренние документы — служебки, заявления, сметы, премии — копились в Word/Excel-файлах
по почте и сетевым папкам. **Кто согласовал?** **Где терпит?** **Сколько ждать?** —
ответы было получить сложно.

ЭДО решает это: **типизированные формы, прозрачные цепочки, аудит каждого шага,
PDF-копия в архиве**.

</div>

## Каталог документов

<br>

| Категория | Типы | Особенности |
|---|---|---|
| 📄 **Служебные записки** | свободная форма, переработка, премирование (×2) | Markdown, табличные премии с автосуммой |
| 📋 **Заявления** | отгул с отработкой, отгул за свой счёт, свободная форма | Личные документы, приватный режим |
| 🔔 **Уведомления** | об отпуске | **Обратный поток**: бухгалтер → сотрудник |
| ✈️ **Командировки** | смета | Авторасчёт, рисованная подпись директора |
| 🏆 **Премирование** | месяц / квартал | Создаёт только руководитель отдела |

> Список расширяется через админ-UI без программирования —
> новый тип = новая запись в каталоге со своими полями и цепочкой.

# Гибкие цепочки

Не только «один за другим»

---

## Параллельные ветки AND / OR

<div class="grid-2">

<div class="card">

### AND-режим

Шаги работают одновременно, нужно одобрение **всех**.

```
Руководитель → [Бухгалтерия || Юрист] → Директор
```

- Любой `reject` → документ закрывается
- Активный inbox у обоих параллельных согласующих

> Удобно когда нужны несколько независимых проверок.

</div>

<div class="card">

### OR-режим

Достаточно **одного** approve, остальные автоматически в `SKIPPED`.

```
[Любой из 3 директоров] → ...
```

- Reject не блокирует — другие могут одобрить
- Документ rejected только если **все** отклонили

> Удобно для дублирующих ролей: «любой из директоров может подписать».

</div>

</div>

---

## Резолверы ролей

Кто согласует — определяется не вручную, а **формулой**.

<br>

| Формула | Что значит |
|---|---|
| `supervisor` | Непосредственный руководитель автора |
| `dept_head_type:division` | Руководитель ближайшего «Дирекция/Управление» вверх по дереву |
| `company_head` | Директор компании (с fallback на корневой Department) |
| `group:accounting@company` | Любой бухгалтер своей компании |
| `field_user_supervisor:employee` | Руководитель сотрудника, указанного в форме |
| `fixed_user:42` | Конкретный человек по ID |
| `author` | Сам автор (для обратных потоков) |

> 11 типов резолверов покрывают любые реальные иерархии — от прямого руководителя
> до «найди ближайший узел нужного типа в дереве подразделений».

# Удобство для пользователя

Каждый день, не раз в год

---

## Email-link одобрение

<br>

**Согласующий получает письмо** с двумя кнопками:

> 📧 **Тема:** Bresler ERP: на согласование — СЗ-СВОБ-2026-0042
>
> Документ ожидает вашего решения. Ваш шаг: Непосредственный руководитель.
>
> ➡ **Согласовать одной кнопкой** → [https://erp.bresler.ru/edo/email-action/abc...]
> ➡ **Отклонить (нужен комментарий)** → [https://erp.bresler.ru/edo/email-action/xyz...]

<br>

- Логин **не требуется** — авторизация по подписанному токену
- Ссылка действительна 14 дней, одноразовая
- Открывается из почты с телефона за 5 секунд

---

## Замещение на отпуск

<br>

<div class="grid-2">

<div>

### Настройка

Сотрудник или администратор задаёт:

- **Замещающий** — коллега, принимающий решения вместо вас
- **Дата начала** замещения
- **Дата окончания** *(необязательная)*

</div>

<div>

### Поведение

Когда наступает период замещения:

- Любой **новый** шаг согласования автоматически уходит замещающему
- Оригинальный approver сохраняется в истории (`original_approver`)
- Уже PENDING-шаги остаются у вас — для них кнопка «Делегировать» вручную

</div>

</div>

> Решает классическую проблему: *«Иванов в отпуске, у него висит 12 документов, как их разблокировать?»*

---

## SLA и эскалация

<br>

Каждый шаг имеет срок (например, 24 часа для руководителя, 72 — для бухгалтерии).

```
1ч  ← Celery Beat сканирует pending-шаги
        ↓
        Если sla_due_at < now → пометить sla_breached_at
        ↓
        Уведомления:
        • автору документа        — bell + email
        • согласующему            — bell + email (напоминание)
        • его руководителю        — эскалация
```

<br>

- Один шаг помечается просроченным **один раз** — без спама
- Все просроченные видны на отчёте админа

---

## Связи между документами

Универсальная привязка двух любых документов через `DocumentLink`:

<br>

| Сценарий | Связь |
|---|---|
| Смета командировки → отчёт по итогам | `travel_estimate` ↔ `memo_free` |
| Служебка о премировании → приказ | `memo_bonus_*` ↔ внешний приказ |
| Заявление на отгул → ответный документ | `app_dayoff_*` ↔ `app_free` |

<br>

- Поиск по номеру или заголовку — связь создаётся в одно действие
- Связь видна с обеих сторон одновременно
- Можно связать с любой моделью системы (заказы, организации, контакты)

# Подпись и архив

Юридическая значимость и audit trail

---

## Рисованная подпись

<br>

Для типов с флагом «требует подпись» (командировка, уведомление об отпуске):

- Согласующий **рисует подпись** на canvas (мышью или пальцем)
- Подпись сохраняется в шаге как data-URL
- В **PDF** автоматически вставляется в подвал документа

<br>

> Это не КЭП и не ЭЦП — но достаточно для внутреннего документооборота
> и юридически защищено логом изменений (simple_history) +
> снепшотом цепочки на момент submit'а.

> КЭП через Диадок/СБИС/КриптоПро — на roadmap для внешних документов.

---

## PDF-экспорт

<br>

В правом верхнем углу страницы документа — кнопка **«PDF»**.

```
┌────────────────────────────────────┐
│  НПП БРЕСЛЕР                       │  ← шапка из справочника
│  Управляющий ГК                    │     «Шапки организаций»
│  Ефимов М. Н.                      │
│                                    │
│  СЗ-ПЕР-2026-0042 от 26.04.2026   │
│                                    │
│  Прошу разрешить выход работников  │  ← отрендеренное тело
│  Сектора 6-35 кВ в выходной день  │     по DTL-шаблону
│  ...                               │
│                                    │
│                  ___[ подпись ]___  │  ← canvas signature
│                  Терентьев А. Ю.   │
└────────────────────────────────────┘
```

- Кешируется на 7 дней, авто-перегенерация при изменениях
- ZIP-архив всех документов за период (для аудита)

# Для администратора

Без программирования

---

## Конструктор типов документов

<br>

Админ создаёт **новые типы документов через UI**:

1. **Поля формы** (`field_schema`) — текст, число, дата, период, выбор, сотрудник, **таблица** с колонками
2. **Шаблоны** заголовка и тела (Django Template Language)
3. **Цепочка согласования** — drag-and-drop шаги с пресетами для role_key
4. **Параметры**: видимость, нумерация, требует ли подпись, кто может создавать

<br>

> **Изменения не ретроактивны:** уже отправленные документы хранят
> снепшот цепочки и шаблона на момент submit'а — смена настроек не
> ломает архив.

---

## Отчёты и массовые операции

<br>

| Отчёт | Что показывает |
|---|---|
| 🕒 **Висящие документы** | Pending дольше N дней — массовое напоминание / отмена |
| ⚠️ **Нарушения SLA** | Шаги с зафиксированной просрочкой за период |
| 📊 **Топ типов** | Гистограмма создаваемых документов — нагрузка на систему |

<br>

**Bulk-операции прямо из отчёта**:

- Чекбоксы → **«Напомнить»** — bell-уведомление текущим согласующим
- **«Отменить»** с обязательной причиной — даже после первого approve
- Идемпотентно: уже закрытые документы безопасно пропускаются

<br>

**ZIP-экспорт архива** за период — `index.json` + PDF + вложения по каждому документу.

## Архитектурные акценты

<div class="grid-2">

<div class="card">

### Snapshot-первая модель

`chain_snapshot`, `body_rendered`, `header_snapshot` фиксируются на submit.
Смена шаблона задним числом — не ломает архив.

<span class="pill">audit-grade</span>

</div>

<div class="card">

### Event-driven

Любое значимое действие — событие через шину. Уведомления, эскалации, кеши — отдельные обработчики, не ломают друг друга.

<span class="pill">composable</span>

</div>

<div class="card">

### Multi-tenant из коробки

Группа компаний? `group:accounting@company` ограничит запрос своей компанией. `tenancy_override` позволяет per-type исключения.

<span class="pill">scalable</span>

</div>

<div class="card">

### Idempotent + observable

Все long-running операции (SLA-чекер, PDF-кеш) идемпотентны. Diagnostic-команды для тикетов.

<span class="pill">production-ready</span>

</div>

</div>
