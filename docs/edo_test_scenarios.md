# ЭДО — сценарии ручного тестирования из UI

Набор для проверки модуля внутреннего документооборота с фронтенда.
Сгруппирован по ролям, чтобы легко делегировать прогон. Каждый сценарий —
шаги + ожидаемый результат + что проверить дополнительно.

> Перед прогоном убедитесь, что в системе есть как минимум:
> - 1 OrgUnit с `business_role=internal, unit_type=company` (например «Релесофт»);
> - дерево Department внутри (минимум 1 уровень с детьми);
> - 1 запись в справочнике «Шапки организаций» (см. сценарий A1);
> - пользователи: обычный сотрудник, руководитель сектора, руководитель
>   отдела (выше сектора), директор компании, бухгалтер, админ.

## A. Подготовка справочников

### A1. Создать запись в «Шапки организаций»

**Роль:** admin
**Откуда:** Сайдбар → ЭДО → Администрирование → Шапки организаций

1. Кнопка **Добавить**.
2. Выбрать организацию (компания с business_role=internal).
3. Ввести ФИО директора и должность.
4. **Действует с** = сегодня; **Действует по** оставить пустым.
5. Кнопка **Сохранить**.

**Ожидаемо:**
- Запись появляется в таблице ниже.
- При создании любого документа далее в PDF в шапке появится этот директор.

### A2. Проверить активного руководителя на дату

**Роль:** admin

1. Создать вторую запись для той же организации с from_date = завтра, ФИО другое.
2. Создать документ сегодня (через сценарий B1).
3. Скачать PDF.

**Ожидаемо:** в шапке PDF — первый директор (актуальный на сегодня), не второй.

## B. Обычный сотрудник

### B1. Создать служебку в свободной форме (минимальный happy path)

**Роль:** обычный сотрудник
**Откуда:** Сайдбар → ЭДО → Создать документ

1. Кликнуть карточку **«Служебная записка (свободная форма)»**.
2. Заполнить:
   - Тема: «Тест служебки B1»
   - Адресовано подразделению: любое, у которого есть руководитель
   - Текст: «Тестовый текст»
3. Снизу — preview цепочки (2 шага).
4. Нажать **«Отправить на согласование»**.

**Ожидаемо:**
- Toast «Документ отправлен на согласование».
- Редирект на страницу документа.
- Статус: «На согласовании» (badge).
- Номер: `СЗ-СВОБ-2026-NNNN`.
- Слева отрендеренное тело (текст с переносами строк).
- Справа в степпере: первый шаг pending (часы), второй waiting (серый).
- В таймлайне: «создан» → «отправлен».

### B2. Заполнить черновик и вернуться к нему

1. Создать документ типа `app_dayoff_unpaid` (заявление на отгул за свой счёт).
2. Заполнить только часть полей (без обязательных).
3. Нажать **«Сохранить черновик»**.

**Ожидаемо:**
- Toast «Черновик сохранён».
- Редирект на страницу документа со статусом «Черновик».
- Номер пустой (не присвоен).
- Перейти в «Мои документы → Черновики» — документ в списке.
- Открыть его → нажать **«Внести правки»** → дозаполнить → **«Отправить»**.
- Статус становится «На согласовании», номер присвоен.

### B3. Заявление в свободной форме конкретному сотруднику

1. Создать `app_free`.
2. Тема: «Тест B3»; Адресат: выбрать конкретного руководителя, не своего.
3. Текст: любой.
4. Submit.

**Ожидаемо:**
- Цепочка из 2 шагов: ваш руководитель → выбранный адресат.
- Если адресат = ваш руководитель — цепочка дедуплицируется до 1 шага.

### B4. Отгул с отработкой

1. Создать `app_dayoff_workoff`.
2. Дата отгула: завтра; дата отработки: послезавтра.
3. Submit.

**Ожидаемо:**
- 2 шага: руководитель → бухгалтерия (`inform`).
- После approve руководителя — статус сразу APPROVED, шаг бухгалтерии
  автоматически в SKIPPED-зелёный (auto-completed).

### B5. Отменить документ до первого approve

1. После B1 — открыть документ, не дожидаясь решения руководителя.
2. Нажать **«Отменить»** в шапке справа.

**Ожидаемо:**
- Toast «Документ отменён».
- Статус: «Отменено».
- Все pending/waiting шаги становятся SKIPPED.
- Закрытие нельзя отозвать — документ навсегда в Архиве.

### B6. Запрошены правки → переотправка

1. Подать любую служебку (например, B1).
2. Зайти под руководителем → нажать **«Запросить правки»** с комментарием
   «Поправь тему, не понятно».
3. Вернуться под автора. На вкладке «Мои в работе» — документ со статусом
   «Запрошены правки» (амбер бейдж).
4. Открыть → **«Внести правки»** → поправить тему → **«Отправить»**.

**Ожидаемо:**
- Документ снова в `pending`.
- В таймлайне видны: submit → revision_requested → submit (повторный).
- Шаги пересоздаются заново (в степпере новый комплект шагов).

## C. Руководитель подразделения

### C1. Премирование месяц с автокалькуляцией total

**Роль:** руководитель сектора (`is_department_head=True`)

1. Создать `memo_bonus_monthly`.
2. Месяц премирования: текущий.
3. В таблицу «Список премий» добавить 3 строки:
   - сотрудник 1, сумма 5000, обоснование «Качественная работа»
   - сотрудник 2, сумма 3000, обоснование «»
   - сотрудник 3, сумма 7000, обоснование «»
4. Проверить, что **«Итого»** автоматически = 15000.
5. Удалить вторую строку. Проверить total = 12000.
6. Submit.

**Ожидаемо:**
- В body PDF — список с фамилиями + суммами + обоснованиями.
- Цепочка: рук → бухгалтерия → директор.
- Документ виден сотрудникам того же подразделения (visibility=`department_visible`).

### C2. Проверка прав инициатора

**Роль:** обычный сотрудник (не руководитель)

1. Зайти в каталог `/edo/new`.

**Ожидаемо:** карточек `memo_bonus_monthly` и `memo_bonus_quarterly` **нет**.
(Они доступны только `is_department_head=True`.)

### C3. Премирование квартал

1. Создать `memo_bonus_quarterly`.
2. Период: «I квартал», Год: 2026.
3. Заполнить таблицу.
4. Submit.

**Ожидаемо:**
- Заголовок документа: «Премирование I квартал 2026».
- В body — фраза «по итогам I квартал 2026 года».
- Резолверы и цепочка как в C1.

### C4. Сверхурочка в выходной с переработкой

1. Создать `memo_overtime`.
2. Характер работ: «В выходной день»; дата: ближайшая суббота; время: 09:00–18:00.
3. Ответственный за ОТ: один из сотрудников подразделения.
4. Список сотрудников: 2-3 человека.
5. Submit.

**Ожидаемо:**
- Body содержит «в выходной день» (не «в сверхурочное время»).
- 3 шага: рук → рук родительского подразделения → бухгалтерия (одобрение).
- Список сотрудников — нумерованный, с инициалами и должностями.

## D. Бухгалтерия

### D1. Уведомление об отпуске (обратный поток)

**Роль:** пользователь в группе `accounting`

1. Создать `vacation_notification`.
2. Сотрудник: выбрать обычного сотрудника.
3. Дата начала: через 2 недели; продолжительность: 14; вид: Ежегодный.
4. Submit.

**Ожидаемо:**
- 3 шага в степпере:
  1. Бухгалтер (роль `author`) — pending.
  2. Сотрудник (`field_user:employee`) — waiting.
  3. Руководитель сотрудника (`field_user_supervisor:employee`) — waiting (action=inform).
5. Бухгалтер открывает документ → видит панель **«Ваше решение»** →
   ставит рисованную подпись на холсте → **«Согласовать»**.
6. Шаг 1 → APPROVED. Шаг 2 → PENDING.
7. Зайти под выбранным сотрудником → у него в «Ждут меня» появляется
   документ → открывает → видит SignaturePad → расписывается → approve.

**Ожидаемо после шага 7:**
- Шаг 2 → APPROVED.
- Шаг 3 (руководитель сотрудника, inform) автоматически → APPROVED.
- Документ в статусе APPROVED.
- В таймлайне 4 события: submit → approve(шаг 1) → approve(шаг 2) → closed.

### D2. Бухгалтер не может выбрать произвольный документ за сотрудника

**Роль:** обычный сотрудник (НЕ в accounting)

1. Зайти в каталог `/edo/new`.

**Ожидаемо:** карточки `vacation_notification` НЕТ — этот тип
доступен только `group:accounting`.

### D3. Group-inbox: один документ на всех бухгалтеров

**Роль:** два пользователя в группе `accounting`

1. Завести 2 бухгалтеров (бухА и бухБ).
2. Под обычным автором отправить `app_dayoff_unpaid`.
3. Руководитель approves → шаг 2 (бухгалтерия) активируется.
4. Зайти как бухА → «Ждут меня» → документ есть.
5. Зайти как бухБ → «Ждут меня» → документ **тоже есть** (group-inbox).
6. БухА approves.

**Ожидаемо:**
- Документ APPROVED.
- На странице документа в шаге 2 — `approver` = бухА (тот, кто кликнул).
- В `original_approver` — pre-resolved бухгалтер (мог быть любой из них).

## E. Согласующий

### E1. Approve без комментария

**Роль:** руководитель сектора, ему пришёл pending-шаг

1. Зайти в «Ждут меня». Открыть документ.
2. В правой колонке — карточка «Ваше решение» (с цветной рамкой primary).
3. Не вводя комментарий — нажать **«Согласовать»**.

**Ожидаемо:**
- Toast «Согласовано».
- Шаг → APPROVED, decided_at = сейчас.
- Документ либо переходит к следующему шагу, либо закрывается.

### E2. Reject требует комментарий

1. Открыть документ ожидающий вашего решения.
2. Не вводя комментарий — нажать **«Отклонить»**.

**Ожидаемо:** Toast «Нужен комментарий», действие не выполнено.

3. Ввести «Не одобрено в этой формулировке» → **«Отклонить»**.

**Ожидаемо:**
- Документ → REJECTED.
- Шаг сохраняет комментарий.
- Автор получает bell + email уведомление с текстом комментария.

### E3. Делегирование

1. Открыть pending-документ.
2. Кнопка **«Делегировать»** в панели «Ваше решение».
3. В диалоге выбрать другого сотрудника.

**Ожидаемо:**
- Toast о делегировании.
- На странице документа в шаге: `approver` = новый, `original_approver` =
  вы.
- Делегату приходит уведомление.
- Кнопки approve/reject у вас исчезают (вы больше не текущий approver).

### E4. Подпись для travel_estimate

**Роль:** директор компании (`is_department_head=True, department_unit=NULL`)

1. Сотрудник создаёт `travel_estimate`, цепочка проходит до директора.
2. Директор открывает → в карточке «Ваше решение» появляется холст подписи.
3. Не подписав — нажать **«Согласовать»**.

**Ожидаемо:** Toast «Документ требует подпись — поставьте её на холсте».
Действие не выполнено.

4. Поставить подпись (нарисовать мышью) → **«Согласовать»**.

**Ожидаемо:**
- Шаг APPROVED, документ APPROVED.
- В шаге сохранена подпись (data:image/png).
- При скачивании PDF подпись отображается в подвале.

## F. Параллельные ветки и продвинутые цепочки

> **Подготовка:** админ создаёт нестандартный тип документа с
> параллельной веткой через `/edo/admin/types/new`.

### F1. AND-параллель: оба должны одобрить

**Роль:** admin (подготовка)

1. Создать тип `test_parallel_and` с цепочкой:
   - Шаг 1: supervisor, action=approve, sla_hours=24
   - Шаг 2: group:accounting@company, action=approve,
     parallel_group="review", parallel_mode="and"
   - Шаг 3: group:legal@company, action=approve,
     parallel_group="review", parallel_mode="and"
2. Завести группу `legal` и положить туда юриста.

**Сценарий:**
1. Сотрудник создаёт документ → submit.
2. Шаг 1 (рук) approves.
3. Открыть как бухгалтер → видит документ в «Ждут меня». Approve.

**Ожидаемо:**
- Шаг 2 → APPROVED. Шаг 3 (юрист) ещё PENDING. Документ в `pending`.

4. Открыть как юрист → approve.

**Ожидаемо:** документ APPROVED.

### F2. AND-параллель: один отклонил → документ rejected

1. То же что F1, но юрист на шаге 3 нажимает **«Отклонить»**.

**Ожидаемо:**
- Документ сразу REJECTED.
- В шагах: бух APPROVED, юрист REJECTED.
- Автор получает уведомление с комментарием юриста.

### F3. OR-параллель: первый approve → остальные SKIPPED

**Подготовка:** изменить parallel_mode="or" в той же цепочке.

1. Сотрудник создаёт. Submit.
2. Approve рук → активируется параллель из 2 шагов.
3. Бухгалтер approves.

**Ожидаемо:**
- Шаг бухгалтера → APPROVED, шаг юриста → **SKIPPED** (автоматически).
- Документ APPROVED.

### F4. OR-параллель: один отклонил → документ продолжает работу

1. Та же подготовка (parallel_mode="or").
2. Юрист отклоняет.

**Ожидаемо:**
- Документ остаётся в `pending`.
- Шаг юриста REJECTED, шаг бухгалтера ещё PENDING.

3. Бухгалтер approves.

**Ожидаемо:** документ APPROVED.

### F5. OR-параллель: все отклонили → rejected

1. То же.
2. Бухгалтер reject, юрист reject.

**Ожидаемо:** документ REJECTED.

## G. Email-link approve

> **Подготовка:** в `.env.local` или `config/settings/development.py`
> установить `EMAIL_BACKEND=django.core.mail.backends.console.EmailBackend`
> и `SITE_URL=http://localhost:5173`. В development email уйдёт в stdout
> Django-сервера — оттуда возьмём ссылку.

### G1. Approve через email

1. Сотрудник отправляет любую служебку (B1).
2. В консоли backend сервера найти email с темой
   «Bresler ERP: на согласование — …».
3. Скопировать URL approve (вида `http://localhost:5173/edo/email-action/<token>/`).
4. Открыть в **отдельной приватной вкладке без логина** — должна открыться
   страница превью.

**Ожидаемо:**
- Видны: тип документа, заголовок, имя автора, имя согласующего, текущий
  шаг.
- Кнопка action.
5. Нажать approve.

**Ожидаемо:**
- Запрос вернёт 200.
- В системе документ переведён на следующий шаг (или закрыт).

### G2. Повторное использование ссылки

1. Использованную в G1 ссылку открыть ещё раз.

**Ожидаемо:**
- Возвращает 400 «Шаг уже закрыт».

### G3. Reject через email требует комментарий

1. Аналогично G1, но для reject-ссылки.
2. POST без комментария → 400.
3. POST с комментарием → 200, документ REJECTED.

### G4. Тампер токена

1. Скопировать reject-URL, дописать в конце «x».
2. Открыть.

**Ожидаемо:** 400 «Ссылка недействительна: invalid signature».

## H. Замещение на время отпуска

### H1. Включить замещение и проверить redirect нового шага

**Роль:** руководитель A (имеет подчинённых) и его коллега B

1. Под A: открыть Профиль → Замещение → выбрать B как замещающего, период:
   с сегодня по +7 дней. Сохранить.
2. Под обычным сотрудником из подразделения A: создать служебку B1.
3. Submit.

**Ожидаемо:**
- В цепочке шаг 1 — pending, approver = B (замещающий, не A).
- В шаге сохранён `original_approver` = A (виден на странице документа,
  если развернуть карточку шага).
- B видит документ в «Ждут меня».
- A в «Ждут меня» этого документа НЕ видит.

### H2. Замещение в будущем — не активно сегодня

1. Установить замещение с from_date = +5 дней.
2. Создать документ от сотрудника A → submit.

**Ожидаемо:**
- approver = A (не замещённый).

### H3. Уже PENDING шаг при включении замещения не перерезолвится

1. Сотрудник отправил документ → шаг ушёл к A (без замещения).
2. A включает замещение.
3. Зайти под B → документа в «Ждут меня» **нет**.

**Ожидаемо:** замещение применяется только к **новым** активациям шагов.
Для уже открытых — A должен использовать кнопку «Делегировать» вручную.

## I. Администратор: типы документов

### I1. Создать новый тип через визуальный редактор

**Роль:** admin
**Откуда:** Сайдбар → ЭДО → Администрирование → Типы документов

1. Кнопка **Создать тип**.
2. Заполнить:
   - Код: `test_simple_memo`
   - Название: «Тестовая простая служебка»
   - Категория: Служебные записки
   - Иконка: `file-text`
   - Нумерация: выбрать любой существующий NumberSequence
   - Видимость: personal_only
   - Кто может: author
3. **Поля:** добавить «subject» (text, required), «body» (markdown, required).
4. **Шаблоны:**
   - title: `Тестовая «{{ subject }}»`
   - body: `{{ body|linebreaks }}`
5. **Цепочка:** добавить 1 шаг, role_key=supervisor, action=approve, sla=24.
6. Сохранить.

**Ожидаемо:**
- Toast «Тип сохранён».
- Редирект на редактирование (теперь `code` заблокирован).
- В каталоге пользователя `/edo/new` появилась карточка нового типа.
- Создание документа этого типа работает.

### I2. Изменение type не влияет на старые документы

1. После I1 — создать документ нового типа.
2. Submit.
3. Вернуться в админ → редактировать тип: добавить ещё один шаг в цепочку.
4. Сохранить.
5. Открыть отправленный ранее документ.

**Ожидаемо:**
- В степпере — старая цепочка из 1 шага (chain_snapshot).
- Новая цепочка применится только к **новым** submit.

### I3. Поле type=table

1. Создать тип с полем «items» (type=table) с колонками:
   - product (text)
   - qty (number)
   - price (money)
2. Шаблон body:
   ```
   {% for row in items %}
   {{ forloop.counter }}. {{ row.product }} — {{ row.qty }}×{{ row.price|floatformat:2 }} ₽
   {% endfor %}
   ```
3. Сохранить → создать документ → заполнить таблицу 2 строки → submit.

**Ожидаемо:** body отрендерен корректно с нумерацией строк.

### I4. Невалидная схема даёт ошибку

1. Создание типа: задать поле с типом «no_such_type».
2. Сохранить.

**Ожидаемо:** toast с ошибкой валидации (`field_schema: ...`).

### I5. Параллельная ветка через UI

1. Создать/редактировать тип → в цепочке добавить 2 шага:
   - оба с parallel_group="check"
   - parallel_mode="and"
2. Сохранить → создать документ → проверить scenario F1.

## J. Администратор: bulk-операции и архив

### J1. ZIP-экспорт архива за период

**Роль:** admin
**Откуда:** Сайдбар → ЭДО → Администрирование → Отчёты

1. В карточке «Экспорт архива (ZIP)» выбрать период (последний месяц).
2. Нажать **Скачать ZIP**.

**Ожидаемо:**
- Toast «Архив скачан (total=N; pdf_ok=M; pdf_failed=K; attachments=A)».
- В Downloads — файл `edo_archive_<from>_<to>.zip`.
- Распаковать: `index.json` + папки на каждый документ + в каждой
  `metadata.json` + `document.pdf` (если есть body) + `attachments/`.

### J2. Bulk-напоминание

> **Подготовка:** иметь несколько документов в pending дольше 3 дней.
> Если их нет — обновите `submitted_at` в Django shell:
> ```python
> from datetime import timedelta
> from django.utils import timezone
> from apps.edo.internal_docs.models import Document
> Document.objects.filter(status="pending").update(
>     submitted_at=timezone.now() - timedelta(days=5))
> ```

1. На странице отчётов в секции «Висящие документы» отметить чекбоксы у 2+ строк.
2. В верхней панели появилась плашка «Выбрано: N» с кнопками **Напомнить** / **Отменить**.
3. Нажать **Напомнить** → диалог.
4. Опционально ввести сообщение → **Отправить напоминание**.

**Ожидаемо:**
- Toast «Напоминания отправлены: N».
- Согласующим этих документов приходят bell-уведомления (категория warning).

### J3. Bulk-отмена с reason

1. Отметить ≥1 строку. Нажать **Отменить**.
2. В диалоге — поле «Причина отмены» обязательно. Не вводя — нажать
   «Отменить документы».

**Ожидаемо:** Toast «Укажите причину», действие не выполнено.

3. Ввести причину «Дубликат» → **Отменить документы**.

**Ожидаемо:**
- Toast «Отменено: N; пропущено: K».
- Документы перешли в статус `cancelled`.
- В шагах в комментарии: «[Отменено администратором ИИИ ИИ.: Дубликат]».
- При попытке отменить уже закрытые — они идут в `skipped`, а не `cancelled`.

### J4. Reports refresh после bulk

1. После J3 обновить страницу.

**Ожидаемо:** отменённых документов больше нет в «Висящие документы»
(они в Архиве).

## K. Edge cases и регрессии

### K1. Документ без резолвимого supervisor

**Подготовка:** создать пользователя без `User.supervisor`, не имеющий
head'а в своём подразделении.

1. Под этим пользователем создать `app_dayoff_workoff`.
2. Submit.

**Ожидаемо:**
- Toast 400 «Не удалось собрать цепочку согласования: Required step
  order=1 role='supervisor' could not be resolved for author N».
- Документ остаётся в draft.

### K2. Дедупликация в цепочке

**Подготовка:** тип, где первые 2 шага по разным role_key, но
резолвятся в одного и того же User.

1. Submit.

**Ожидаемо:**
- В шагах документа — 1 шаг (один из дублей пропущен с логом).

### K3. Inform-шаг между active

**Подготовка:** тип с цепочкой supervisor → group:accounting (inform) → company_head.

1. Submit. Approve рук.

**Ожидаемо:**
- Шаг 2 (inform) сразу → APPROVED.
- Шаг 3 (директор) → PENDING.

### K4. Visibility=`personal_only` для коллеги

1. Создать `app_dayoff_unpaid` (visibility=personal_only). Submit.
2. Зайти под коллегой из того же подразделения, который **не** в цепочке.
3. Открыть документ напрямую по URL.

**Ожидаемо:** «Документ не найден или у вас нет доступа».

### K5. Visibility=`department_visible`

1. Создать `memo_overtime` (visibility=department_visible). Submit.
2. Под коллегой из того же подразделения (не согласующий) — зайти на /edo/my.

**Ожидаемо:** документ доступен в Архиве после закрытия (или в каком-то
из табов, если покрывает текущую логику).

### K6. PDF-кеш инвалидируется при изменении

1. Открыть документ → скачать PDF — запомнить размер и текст.
2. Под автором (если revision_requested) внести правку → submit.
3. Снова PDF.

**Ожидаемо:** новый PDF, текст обновлён.

### K7. SLA breach (требует Celery worker)

> **Только если у вас запущен Celery beat + worker.**

1. Создать документ с `sla_hours=1` в цепочке (тип с короткой SLA — нужно
   создать через админку).
2. Не одобрять. Подождать ≥1 час.
3. Вызвать вручную: `manage.py shell` → `from apps.edo.internal_docs.tasks
   import check_sla_breaches; check_sla_breaches()`.
4. Зайти как автор → bell-уведомления.
5. Зайти как согласующий → bell-уведомления.
6. Зайти как руководитель согласующего → bell-уведомления (эскалация).

**Ожидаемо:** в каждом аккаунте — соответствующее уведомление в категории
warning. Повторный запуск задачи не плодит дубли (`sla_breached_at`
проверяется).

### K8. Замещение + параллельная ветка

1. У одного из двух параллельных согласующих включить замещение.
2. Submit нового документа.

**Ожидаемо:** в pending-шаге у замещаемого approver = замещающий, у
второго — он сам. Оба шага активные одновременно.

### K9. Связь между двумя документами

1. Создать документ A. Submit.
2. Создать документ B. Submit.
3. На странице A в карточке «Связанные документы» нажать **Связать**.
4. В поиске ввести часть номера документа B.
5. Кликнуть найденный B.

**Ожидаемо:**
- В обеих документах (A и B) появляется связь, ссылка с обеих сторон.
- Удалить связь со страницы A — она пропадает у обоих.

### K10. Self-link и дубль

1. На странице документа A в LinkPicker найти A.

**Ожидаемо:** при попытке создать связь — toast «Нельзя связать документ
сам с собой».

2. Создать связь A↔B (см. K9).
3. Попытаться создать вторую такую же.

**Ожидаемо:** toast «Связь уже существует».
