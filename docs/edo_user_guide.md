# ЭДО — пользовательский гайд

Внутренний документооборот: служебные записки, заявления, уведомления,
командировочные сметы. Все документы создаются через типизированные формы,
проходят цепочку согласования и архивируются с PDF-копией.

> 🎯 **Краткий обзор для ознакомления** — [edo_presentation.md](edo_presentation.md) (~22 слайда).
> 📚 **Если ты администратор** — [edo_admin_guide.md](edo_admin_guide.md).

## 1. Что и где находится

**Меню → ЭДО** (если в твоей роли есть доступ к модулю edo).

Внутри сразу 4 вкладки:

| Вкладка | Что показывает |
|---|---|
| **Ждут меня** | Документы, по которым **от тебя** требуется решение. С красной плашкой-счётчиком, когда есть свежие. Обновляется каждую минуту. |
| **Мои в работе** | Что **ты** отправил, но цепочка ещё не закрыта. |
| **Черновики** | Заполненные тобой документы без отправки на согласование. Номер не присваивается до submit. |
| **Архив** | Закрытые документы (`approved` / `rejected` / `cancelled`) — твои и где ты участвовал. |

Дополнительно в сайдбаре есть прямые ссылки **Входящие на согласование**,
**Мои отправленные**, **Черновики**, **Архив** — это shortcut'ы к тем же
вкладкам.

## 2. Создать документ — общий wizard

1. Кнопка **Создать** в правом верхнем углу страницы «Документооборот».
2. Откроется **каталог типов**, сгруппированный по категориям:
   - Служебные записки
   - Заявления
   - Уведомления
   - Командировки
   - Премирование
3. Кликаешь нужный тип — попадаешь на форму создания.

### Заполнение формы

Поля динамически рендерятся под тип документа. Поддерживаются:

| Тип поля | Как выглядит | Особенности |
|---|---|---|
| Текст | строка | |
| Многострочный текст / Markdown | textarea | Переводы строк сохраняются |
| Число / Сумма (₽) | поле с шагом | Money — два знака после запятой |
| Дата | date-picker | |
| Период (от–до) | две даты в строке | |
| Время | time-picker | |
| Да/Нет | checkbox | |
| Выбор из списка | select | |
| Сотрудник | поиск по имени | Возвращает FK на User |
| Несколько сотрудников | multi-select | |
| Организация / Подразделение | поиск | |
| Файл | upload | |
| **Таблица** | многострочная сетка | Кнопки «Добавить строку» / «Удалить» |

Под формой — справочный блок **«Цепочка согласования»**: показывает, кто за
кем будет смотреть документ, с указанием ролей и SLA. Это статичный
preview — реальные согласующие резолвятся при отправке (учитывают
замещения, дерево подразделений, групповые роли).

### Внизу страницы — две кнопки

- **Сохранить черновик** — документ остаётся на вкладке «Черновики», номер
  не присваивается. Можно вернуться и доделать позже.
- **Отправить на согласование** — присваивается номер (`СЗ-СВОБ-2026-0042`
  и т.п.), документ уходит первому в цепочке. После этого редактирование
  невозможно (только если согласующий вернёт его в `revision_requested`).

### Автокалькуляции

Для двух типов часть полей считается автоматически:
- **`memo_bonus_*`** — `total` = сумма всех `amount` в таблице.
- **`travel_estimate`** — `total` = `transport_cost` + `lodging_cost` + `per_diem`.

Эти поля можно править вручную (например, если суточные нестандартные),
но при следующем изменении источников они снова пересчитаются.

## 3. Описание всех 9 типов документов

### 3.1 Служебная записка (свободная форма)

**Код:** `memo_free` · **Категория:** Служебные записки · **Иконка:** 📄

**Когда использовать.** Любая внутренняя записка в подразделение по
произвольному вопросу: «Прошу выделить кабинет», «Прошу командировать на
семинар» и т.п.

**Поля:**
- Тема (text)
- Адресовано подразделению (department-picker)
- Текст записки (markdown)

**Цепочка:** Непосредственный руководитель → Руководитель адресата

> Если адресат совпадает с твоим руководителем — второй шаг
> автоматически дедуплицируется (один и тот же человек одобряет один раз).

### 3.2 Служебная записка на переработку

**Код:** `memo_overtime` · **Иконка:** ⏰⚠️

**Когда.** Согласование выхода сотрудников в сверхурочное время или в
выходной день.

**Поля:**
- Характер работ (выбор: сверхурочно / в выходной)
- Дата выхода
- Время с / Время до
- Ответственный за ОТ (выбор сотрудника подразделения)
- Список сотрудников (multi-select)
- Обоснование (опционально)

**Цепочка:** Руководитель → Руководитель родительского подразделения →
Бухгалтерия (одобрение, не inform)

> Создаёт **только руководитель подразделения** (`is_department_head=True`).

### 3.3 Служебная записка на ежемесячное премирование

**Код:** `memo_bonus_monthly` · **Категория:** Премирование · **Иконка:** 📈

**Когда.** Премирование сотрудников по итогам месяца.

**Поля:**
- Месяц премирования (любая дата внутри месяца — берётся month/year)
- **Список премий** (таблица): сотрудник + сумма + обоснование. Кнопка
  «Добавить строку», удаление иконкой корзины.
- Итого (money) — пересчитывается автоматически.

**Цепочка:** Руководитель → Бухгалтерия → Директор

> Создаёт только руководитель подразделения. Видим всем сотрудникам
> подразделения автора (visibility=`department_visible`).

### 3.4 Служебная записка на квартальное премирование

**Код:** `memo_bonus_quarterly` · **Иконка:** 🏆

То же, что 3.3, но с полем «Период» (Q1/Q2/Q3/Q4/«По итогам года») и
«Год».

### 3.5 Заявление на отгул с отработкой

**Код:** `app_dayoff_workoff` · **Категория:** Заявления · **Иконка:** 🔁

**Когда.** Когда нужен один день отгула с обещанием отработать.

**Поля:**
- Дата отгула
- Дата отработки
- Причина (опционально)

**Цепочка:** Руководитель → Бухгалтерия (`inform`, без блокировки —
автоматически закроется при approve руководителя)

### 3.6 Заявление на отгул за свой счёт

**Код:** `app_dayoff_unpaid` · **Иконка:** 🗓✖

**Когда.** Отпуск без сохранения зарплаты на 1 или несколько дней.

**Поля:**
- Период отгула (от–до)
- Причина (обязательно)

**Цепочка:** Руководитель → Бухгалтерия (`approve`, не inform — она реально
проверяет и подписывает).

### 3.7 Заявление в свободной форме

**Код:** `app_free` · **Иконка:** ✉

**Когда.** Адресное заявление конкретному сотруднику (запрос перевода,
обращение к директору и т.п.).

**Поля:**
- Тема
- Кому адресовано (выбор сотрудника)
- Текст (markdown)

**Цепочка:** Руководитель → Адресат заявления

### 3.8 Уведомление об отпуске

**Код:** `vacation_notification` · **Категория:** Уведомления · **Иконка:** 🔔

**ВНИМАНИЕ: обратный поток.** Создаёт **только бухгалтер**, подписывают
последовательно — бухгалтер, потом сотрудник, в конце руководителю
сотрудника уходит копия «к сведению».

**Поля:**
- Сотрудник (выбор User)
- Дата начала отпуска
- Продолжительность (дней)
- Вид отпуска (ежегодный / дополнительный)

**Цепочка:**
1. Бухгалтер (роль `author`) — подписывает (sign + рисованная подпись).
2. Сотрудник (резолв `field_user:employee`) — подписывает в знак ознакомления.
3. Руководитель сотрудника (резолв `field_user_supervisor:employee`) — `inform`
   (закрывается автоматически).

> Поле «Подпись» — рисованная (требуется на 1 и 2 шагах).

### 3.9 Смета на командировку

**Код:** `travel_estimate` · **Категория:** Командировки · **Иконка:** ✈

**Когда.** Расчёт расходов на служебную командировку.

**Поля:**
- Город назначения
- Цель командировки (textarea)
- Период командировки (от–до)
- Транспорт (₽)
- Проживание (₽)
- Суточные (₽) — *вручную, политика суточных у каждой компании своя*
- Итого (₽) — пересчитывается из транспорт + проживание + суточные
- Запрашивается аванс? (Да/Нет)

**Цепочка:** Руководитель → Бухгалтерия → Директор (sign + рисованная подпись)

## 4. Согласование документа

### Как попасть к решению

Три пути:
1. **Через ЛК** — вкладка «Ждут меня» → клик по строке.
2. **Через bell-уведомление** в правом верхнем углу системы.
3. **Через email** — в каждом письме две action-ссылки (см. §6).

### Страница документа

Слева:
- **Шапка** — номер, тип, статус-бейдж, автор, дата отправки.
- **Тело документа** — отрендеренный текст с подставленными значениями полей.
- **Поля документа** — таблица «значение → значение».
- **Вложения** — список файлов от автора и согласующих.
- **Связанные документы** — другие документы ЭДО, прикреплённые к этому.
- **Комментарии и история** — таймлайн (события: создан, отправлен,
  approve/reject каждого шага).

Справа:
- **Цепочка согласования** — степпер из шагов с иконкой статуса и аватаром.
  Когда шаг pending — иконка часов; approved — зелёная галочка; rejected —
  красный крест.
- **Ваше решение** (если ты — текущий согласующий):
  - Поле комментария (обязательно для reject и revision_requested).
  - Холст для рисованной подписи (если документ её требует).
  - Кнопки: Согласовать / Запросить правки / Отклонить / Делегировать.

### Что значат кнопки

| Кнопка | Что делает | Финально? |
|---|---|---|
| Согласовать | Отметить шаг approved, перейти к следующему | Нет |
| Запросить правки | Документ возвращается автору в статус `revision_requested`, тот правит и снова отправляет | Нет |
| Отклонить | Документ закрывается со статусом `rejected`, повторная отправка невозможна | **Да** |
| Делегировать | Передать решение другому сотруднику | Нет |

### Делегирование

Кнопка «Делегировать» открывает выбор сотрудника. После делегирования:
- Текущий approver = выбранный сотрудник.
- В шаге сохраняется `original_approver` = ты.
- Новому approver уходит уведомление.

Делегировать можно только свои шаги. Делегировать самому себе нельзя.

## 5. Параллельные ветки

Если в цепочке несколько шагов помечены одним `parallel_group`, они
становятся активными одновременно.

**AND (по умолчанию).** Нужно одобрение **всех** в группе. Любой `reject`
закрывает документ как rejected — даже если другие уже одобрили.

**OR.** Достаточно одного `approve`, остальные автоматически отметятся
SKIPPED. Reject в OR-режиме не блокирует — документ продолжит работу.
Документ rejected, только когда **все** в OR-группе отклонили.

> Текущий тип `vacation_notification` использует sequential, без
> параллельностей. Параллельные ветки можно настроить админу при
> создании нестандартных типов через админ-UI.
>
> Также для подбора согласующего по уровню в иерархии админ может
> использовать `dept_head_type:<unit_type>` — например, «руководитель
> дирекции/управления» (`division`), «руководитель службы» (`service`)
> и т.п. Это надёжнее `dept_head:up(N)` при разной глубине у разных
> авторов.

## 6. Email-link согласование (без логина)

В каждом email-уведомлении о новом шаге зашиты две ссылки:
- **Согласовать одной кнопкой**
- **Отклонить (нужен комментарий)**

**Что происходит при клике:** открывается публичная страница вида
`<SITE_URL>/edo/email-action/<token>/`. На ней:
1. Сводная карточка документа: номер, заголовок, автор, твой шаг.
2. Поле комментария — обязательное для reject, опциональное для approve.
3. Кнопка подтверждения действия.
4. После клика — экран успеха со ссылкой «Открыть документ».

Особенности:
- Авторизация **по подписанному токену**, не нужно логиниться.
- TTL ссылки — 14 дней.
- Если шаг уже закрыт (тобой через ЛК или коллегой по делегату) —
  страница покажет «Действие невозможно: шаг уже закрыт».
- Ссылка одноразовая по факту: повторный клик после успешного действия
  тоже выдаст «шаг уже закрыт».

> Это удобно, когда ты в дороге и нужно быстро одобрить. Но
> внимательно — этот канал не показывает тебе тело документа в полном
> объёме (в email только заголовок и сводка). Если документ серьёзный —
> открывай через «Открыть в системе».

## 7. Замещение на время отпуска

> ⚠ На текущий момент UI замещения **в Профиле не реализован**. Настраивает
> администратор через Django admin (`/admin/users/user/<твой_id>/`,
> поля `substitute_user`, `substitute_from`, `substitute_until`).
> Самостоятельная настройка из ЛК — в roadmap.

После настройки администратором доступны:
- Замещающий (кто принимает решения вместо тебя).
- Дата начала замещения.
- Дата окончания (можно оставить пустой — бессрочное).

Когда наступит этот период, любой **новый** шаг согласования, который
должен был активироваться у тебя, автоматически уйдёт замещающему. В
истории сохранится `original_approver = ты`.

> ⚠️ **Уже PENDING-шаги при включении замещения задним числом не
> перерезолвятся.** Это by design — иначе нарушится аудит. Для уже
> открытых шагов используй кнопку «Делегировать» вручную.

## 8. SLA, просрочки и эскалация

Каждый шаг согласования имеет срок (SLA), указанный при создании типа:
например, 24 часа для шагов руководителя, 48 — для директора.

Если шаг висит дольше срока:
1. Раз в час система фиксирует просрочку (помечает `sla_breached_at`).
2. Шлёт уведомления:
   - Тебе как автору — «У такого-то висит мой документ».
   - Согласующему — напоминание.
   - **Его непосредственному руководителю** — для эскалации.

Один шаг помечается просроченным один раз — повторных уведомлений по
тому же шагу не будет.

Просроченные шаги видны на отчёте `/edo/admin/reports` (только админам).

## 9. Связи между документами

На странице любого документа есть карточка **«Связанные документы»**.
Клик «Связать» открывает диалог:
1. Введи ≥2 символов от номера или заголовка целевого документа.
2. Из списка кликни нужный — связь создастся (тип `related`).
3. Связи показываются с двух сторон: и у источника, и у цели.

**Когда использовать.** Привязать смету командировки к отчёту по итогам.
Привязать служебку об уходе к приказу. Привязать заявление к
переводу/ответу.

Удалить связь — иконка корзины при наведении на строку (показывается
только если есть права на редактирование документа).

## 10. Вложения и файлы

Карточка **Вложения** на странице документа.

- **Автор** документа может прикладывать файлы в любой момент (черновик /
  на согласовании / закрытый — кроме случая, когда документ редактировать
  нельзя).
- **Активный согласующий** может прикладывать файлы во время своего шага
  (например, бухгалтерия прикрепляет расчёт). Файл связывается со шагом —
  в истории видно, кто и когда добавил.
- Все участники документа (автор, согласующие, dept_head автора) видят
  список и могут скачать.

## 11. PDF-экспорт и архив

В правом верхнем углу страницы документа — кнопка **PDF**.

Что попадает в PDF:
- Шапка: компания + директор (берётся из справочника «Шапки организаций»
  на дату отправки) + блок «От кого / Кому».
- Заголовок документа + номер + дата.
- Тело (отрендеренное по шаблону).
- Подвал: дата + рисованная подпись (если есть).

PDF кешируется на 7 дней; если ты исправил поля или подписи —
автоматически перегенерируется при следующем запросе.

## 12. Частые сценарии и FAQ

**В: Я отправил документ, но вижу, что цепочка пошла не туда. Что делать?**
О: Если ни один шаг ещё не одобрен — кнопка «Отменить» вернёт документ
в `cancelled`. После — создавай заново. Если уже есть approve — нажми
«Запросить правки» у согласующего, он вернёт тебе документ.

**В: У меня нет руководителя в системе, и `submit` падает с ошибкой
«Не удалось собрать цепочку».**
О: Обратись к администратору — ему нужно либо назначить тебе явного
руководителя через `User.supervisor`, либо проставить кому-то в твоём
подразделении флаг `is_department_head=True`.

**В: Мне пришёл email о согласовании, но я не помню пароль/не могу
зайти. Как одобрить?**
О: В email есть кнопки прямого approve/reject — они работают без логина
по подписанной ссылке. TTL — 14 дней.

**В: Я случайно нажал «Отклонить» вместо «Запросить правки». Можно
откатить?**
О: Нет, reject — финальный статус. Попроси автора создать документ
заново. (Это by design: история должна быть достоверной.)

**В: Можно создать тип документа с большим количеством шагов, чем
типовые?**
О: Через админ-UI (`/edo/admin/types`) администратор может настроить
любую цепочку — от 1 шага до десятков, с параллельными ветками и
inform-шагами в произвольных местах.

**В: Как узнать, кто реально согласовал документ, если шаг был групповым
или делегированным?**
О: На странице документа в степпере справа — карточка шага показывает
**фактического** согласующего и (если был) `original_approver` —
исходно назначенного человека.

**В: Что произойдёт со ссылкой из email, если я её перешлю коллеге?**
О: Коллега сможет её использовать — это уязвимость. Не пересылай ссылки.
Лучше делегируй шаг через ЛК.

## 13. Что делать, если…

| Симптом | Решение |
|---|---|
| Документ создан, но статус «Черновик» и нет номера | Ты не нажал «Отправить». Открой и нажми зелёную кнопку. |
| В каталоге типов нет нужного типа | Либо у тебя нет прав создавать (например, премирование — только для руководителей), либо тип ещё не добавлен админом. |
| После submit ошибка «Не удалось собрать цепочку» | Кто-то в цепочке не резолвится. См. FAQ выше. |
| В шаге не показывается approver, написано «Любой сотрудник группы» | Это коллективный шаг (group). Любой член группы может одобрить — приписки конкретного сотрудника нет, пока кто-то не возьмёт его в работу. |
| Ссылки из email не работают | TTL 14 дней истёк, либо шаг уже закрыт другим путём. Открой документ через ЛК. |
| Не приходят email-уведомления | Проверь в Профиле → Настройки уведомлений: возможно, email отключён по этой категории. |
| Пропала рисованная подпись после edit поля | Подпись сохраняется на конкретном шаге. Если автор отредактировал поля и переотправил документ (re-submit) — все шаги пересоздаются заново, и подписи нужно ставить ещё раз. |
| Не вижу документ, на который меня ставили согласующим | Проверь во вкладке «Архив» — может, шаг уже закрыт через email-link или делегирован. |
